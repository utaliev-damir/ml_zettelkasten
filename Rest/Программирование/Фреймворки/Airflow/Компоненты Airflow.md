#фреймворк #airflow 

[Источник]([Architecture Overview — Airflow Documentation (apache.org)](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/overview.html))

DAG - граф, представление потока задач, связанные между собой в виде зависимостей.

Таск- задача, ее описание хранится в DAG folder
* могут иметь зависимости между собой, они определяются операторами `>>` или методами 
* таск это экземпляр оператора или сенсора
	* сенсор - это шаблон для таска, работающий при выполнении опр. условий
	* оператор - это шаблон для таска, имеющий свое функционально назначение. 
	* все операторы, сенсоры наследуются от BaseOperator - на его основе можно делать кастомные операторы
* между тасками можно передавать данные
	* через Xcom - способ передачи небольших метаданных между тасками
	* через загрузку и выгрузку данных в/из хранилища
	* через Task Flow Api - при создании ДАГов таким способом, данные передаются автоматически (в качестве входных параметров)
	

Airflow состоит из нескольких компонентов
* Основные 
	* Sсheduler - планировщик, отвечает за парсинг DAG folder, триггер executor-а, который запускает выполнение задач и который является атрибутом планировщика.
	* Webserver - веб интерфейс который позволяет управлять запуском дагов, получать информацию о ходе выполнения
	* DAG Folder - папка хранящая файлы с дагами ^0aa8a2
	* Metadata Database - компоненты загружают в БД метаданные - информацию о тасках, расписание и т.д.

* Дополнительные  ^eac50a
	* Workers - воркеры, могут быть сторонним компонентом, воркеры это объекты, которые получают задачу от exrcutor-a.
	* Triggerer - некоторые задачи, могут совершать запрос в сторонний сервис и ожидать ответ, тем самым занимая воркер, но не совершая работы. Triggerer необходим для таких асинхронных задач и запускает их в цикле событий.
	* Dag processor - также может быть вынесен на отдельную машину, представляет собой компонент, которые занимает чтением и социализацией дагов из  DAG Folder, состоит из двух компонентов - manager и processor. 
	* Plugins - дополнительный пакет,  в нем можно создавать дополнительные операторы, расширяющие функционал оркестратора. 

Роли в airflow - в зависимости от уровня доступа и способа развертывания Airflow выделяют три роли  в рамках [[Airflow Security Model]]:  ^c837b0
* Deployment manager
* Dag author
* Operator

Компоненты могут быть [[Способы развёртывания Airflow| развернуты]] как на одной машине, так и на нескольких
