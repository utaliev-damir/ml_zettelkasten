# Хэш-таблица (Hash Table)

## Назначение

Позволяет хранить значения и получать их по ключу, который может быть не только числом, но и строкой или другим объектом.

## Основная идея

- Проблема
    - У нас есть слово, которому соответствует какое-то значение, как организовать хранение таких данных?
    - В массиве значения бы лежали по ячейкам, и получить мы бы их могли только по числовым индексам
    - И нам бы нужно было помнить числовой индекс каждого значения чтобы его получить
- Mapping
    - Мы можем придумать функцию, которая будет переводить слово в число. Нужно чтобы функция всегда возвращала один и тот же результат для одинаковых слов
    - Но такая функция может генерить числа, которые выходят за предел размера массива
    - Тогда мы можем полученное число делить с остатком на размер массива
        - Например "ключевое слово" **→ 982462**, размер массива = 8, делим 982462%8=6
    - Используем результат в качестве индекса и кладем туда значение
        - Когда нам нужно будет его достать, хэш-функция переведет в нужный индекс наш ключ и по нему мы заберем значение
- Expander
    - Что будет если место в массиве закончилось, а нам нужно добавить новый элемент
    - Тогда нам нужно создать массив например в два раза больше первого, для каждого ключа рассчитать новый хэш и индекс, положить туда значения
    - Но если у нас будет очень большой словарь, то добавление даже одного элемента, может привести к долгим вычислениям
- Коллизии
    - Может случиться так, что два разных ключа после хэширования и взятия остатка дадут одинаковый индекс
    - Это называется коллизией, и её нужно как-то разрешать
    - Способы разрешения:
        - **Метод цепочек (chaining)** — в каждой ячейке массива хранится не одно значение, а связный список пар (ключ, значение). При коллизии новый элемент добавляется в список
        - **Открытая адресация (open addressing)** — если ячейка занята, ищем следующую свободную по определённому правилу (линейное пробирование, квадратичное, двойное хэширование)

## Свойства

- **Неупорядоченность** — элементы хранятся в порядке, определяемом хэш-функцией, а не порядком добавления
- **Уникальность ключей** — один ключ соответствует одному значению; повторная вставка перезаписывает старое значение
- **Коэффициент заполнения (load factor)** — отношение количества элементов к размеру массива; обычно при превышении ~0.7 запускается рехэширование
- **Амортизированная константная сложность** — отдельные операции могут быть медленными (при рехэшировании), но в среднем по большому числу операций — O(1)
- **Зависимость от качества хэш-функции** — плохая функция приводит к множеству коллизий и деградации до O(n)
- **Дополнительная память** — обычно требуется массив большего размера, чем количество элементов, чтобы держать load factor низким

## Поддерживаемые операции

- `put(key, value)` / `set(key, value)` — добавить или обновить пару ключ-значение
- `get(key)` — получить значение по ключу
- `remove(key)` / `delete(key)` — удалить пару по ключу
- `contains(key)` / `has(key)` — проверить наличие ключа
- `size()` — получить количество элементов
- `keys()` — получить все ключи
- `values()` — получить все значения
- `clear()` — очистить таблицу

## Сложность операций

|Операция|Временная сложность (среднее)|Временная сложность (худший случай)|Пространственная сложность|
|---|---|---|---|
|Вставка `put`|O(1)|O(n)|O(1)|
|Поиск `get`|O(1)|O(n)|O(1)|
|Удаление `remove`|O(1)|O(n)|O(1)|
|Проверка `contains`|O(1)|O(n)|O(1)|
|Рехэширование|O(n)|O(n)|O(n)|
|Общая память|—|—|O(n)|

> Худший случай O(n) возникает при плохой хэш-функции или специально подобранных данных, когда все ключи попадают в одну ячейку.

## Когда использовать

Признаки задач:

- Нужен быстрый доступ к данным по ключу (имя → телефон, id → объект, слово → перевод)
- Требуется проверять наличие элемента в коллекции за O(1) — например, при поиске дубликатов
- Нужно подсчитать частоту встречаемости элементов (например, частота слов в тексте)
- Нужна группировка данных по какому-то признаку (anagram groups, группировка транзакций по пользователю)
- Кэширование результатов вычислений (memoization)
- Поиск пар, сумма/разность которых равна заданному значению (классика — Two Sum)
- Когда порядок элементов не важен (если важен — рассмотреть LinkedHashMap или аналоги)

**Когда НЕ использовать:**

- Нужен порядок элементов или диапазонные запросы → лучше дерево (TreeMap, BST)
- Требуется минимальное потребление памяти → массив или другая компактная структура
- Очень мало элементов (десятки) → линейный поиск по массиву может быть быстрее за счёт кэша процессора

## Реализация

Основная схема реализации (метод цепочек):

```python
class HashTable:
    def __init__(self, capacity=8):
        self.capacity = capacity
        self.size = 0
        self.buckets = [[] for _ in range(capacity)]
        self.load_factor_threshold = 0.7

    def _hash(self, key):
        return hash(key) % self.capacity

    def put(self, key, value):
        index = self._hash(key)
        bucket = self.buckets[index]

        # Обновление существующего ключа
        for i, (k, v) in enumerate(bucket):
            if k == key:
                bucket[i] = (key, value)
                return

        # Вставка нового ключа
        bucket.append((key, value))
        self.size += 1

        # Проверка необходимости рехэширования
        if self.size / self.capacity > self.load_factor_threshold:
            self._resize()

    def get(self, key):
        index = self._hash(key)
        for k, v in self.buckets[index]:
            if k == key:
                return v
        raise KeyError(key)

    def remove(self, key):
        index = self._hash(key)
        bucket = self.buckets[index]
        for i, (k, v) in enumerate(bucket):
            if k == key:
                bucket.pop(i)
                self.size -= 1
                return
        raise KeyError(key)

    def _resize(self):
        old_buckets = self.buckets
        self.capacity *= 2
        self.buckets = [[] for _ in range(self.capacity)]
        self.size = 0
        for bucket in old_buckets:
            for key, value in bucket:
                self.put(key, value)
```

**Ключевые моменты реализации:**

1. **Хэш-функция** — должна быстро вычисляться и равномерно распределять ключи
2. **Размер массива** — часто берут степень двойки (для быстрого `% n` через битовые операции) или простое число (для лучшего распределения)
3. **Разрешение коллизий** — выбирается между цепочками и открытой адресацией в зависимости от паттерна использования
4. **Рехэширование** — запускается при превышении load factor; новый размер обычно в 2 раза больше
5. **Удаление при открытой адресации** — нельзя просто очищать ячейку, нужны "tombstone" маркеры, иначе сломается поиск
